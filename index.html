ads-ai ведёт кампании в Google Ads и Meta Ads (Facebook, Instagram) — анализирует, готовит каждое решение о бюджете, паузе, объявлении — и не применяет ни одного без вашего явного «да». Один цикл анализа, оба кабинета, общий approval gate.

Как мы работаем

Четыре шага между бюджетом и результатом

1
Анализ
Статистика по кампаниям, ключевым словам, объявлениям и поисковым запросам подтягивается каждый цикл — вручную или по расписанию демона.
2
Предложение
AI формулирует конкретное решение с цифрами и причиной — «поставить на паузу», «поднять бюджет на 20%», «добавить минус-слово», а не общие советы.
3
Ваше «да»
Вручную по одному, разом через approve --all, или заранее — по узкому опциональному правилу. Ничего не применяется без этого шага.
4
Исполнение и след
Решение применяется, при необходимости откатывается одной командой, а само решение и его причина остаются в журнале.
Почему это безопасно

Бюджет клиента — не место для экспериментов

Approval gate — единственный путь к трате бюджета, зашит в код, а не в настройках.
Полный откат — любое применённое решение можно вернуть «как было» одной командой.
Жёсткий потолок дневного бюджета, который система не может обойти сама.
Создание и полное отключение кампании — только вручную, всегда, без исключений.
Автономия — опционально и только для узкого списка низкорисковых решений.
Технология

Один оркестратор, два мокаемых контура, один шлагбаум

Для тех, кому интересно, как это устроено внутри: CLI и демон — один и тот же код, обе внешние зависимости спрятаны за диспетчером с полноценным моком. На всей схеме только одна стрелка умеет тратить деньги.

живой вызов
не подключено или opt-in
требует пропуска через gate

Meta Ads зеркалит эту же схему отдельным блоком — MetaAdsClient рядом с AdsClient, тот же Agent, тот же approval gate. Telegram Bot API — двунаправленный: исходящие уведомления (sendMessage) и, при allow_remote_approve, входящие нажатия кнопок (getUpdates long-polling) — оба ведут в тот же approval gate, что proposals approve/reject.
